# 通用表格智能校验与变更说明生成器 技术实现说明书

## 一、项目目标

本项目的目标，是在一天内完成一个可本地运行、可直接演示的 MVP 工具，用来解决表格版本比对中的两个常见问题：

- 表格本身有没有明显的数据质量问题
- 新旧两版表格到底改了什么，如何快速说明给别人看

围绕这个目标，项目没有追求复杂平台能力，而是优先保证三件事：

1. 普通用户可以直接上手使用
2. 核心分析结果稳定、可解释
3. 演示链路完整，从上传到导出能够跑通

因此，这个项目的定位不是“数据治理平台”或“自动修复系统”，而是一个轻量、清晰、可交付的表格校验与变更说明工具。

## 二、技术选型及原因

### 1. Python

选择 Python，主要是因为它非常适合处理结构化数据，也能在有限时间内快速完成原型。对于表格读取、数据比对、规则校验和文本拼装，Python 都有成熟且稳定的生态。

### 2. Streamlit

前端框架选择 Streamlit，而不是单独开发前后端，原因很直接：

- 能快速搭建可交互页面
- 适合文件上传、结果展示、报告下载这类中后台工具场景
- 对数据分析类项目非常友好，开发成本低

这让项目可以把精力集中在分析逻辑，而不是页面工程化本身。

### 3. pandas

`pandas` 是这个项目的数据处理中枢，负责承接表格读取后的 DataFrame 计算。之所以选择它，是因为：

- 适合做行列级分析
- 适合处理缺失值、类型判断、聚合统计
- 对比新旧表格时，按主键建立映射和提取差异都比较自然

### 4. openpyxl 与 xlrd

项目要求支持 Excel 和 CSV，因此选择：

- `openpyxl` 读取 `.xlsx`
- `xlrd` 兼容 `.xls`

这样可以覆盖用户最常见的本地表格格式，避免把输入范围限制得过窄。

### 5. 为什么没有引入数据库、消息队列或更复杂框架

这是一个本地单机场景的演示项目，主要处理用户临时上传的文件，不涉及长期存储、多人协作、权限体系或异步任务管理。此时如果引入数据库或更复杂架构，收益很小，反而会稀释 MVP 的完成度。

本项目有意保持“轻后端、重核心逻辑”的实现方式，符合题目约束，也更适合作为面试提交材料。

## 三、整体架构

项目采用的是很典型的“单入口页面 + 模块化分析逻辑”结构。

### 1. 架构思路

- `app.py` 负责页面交互、状态管理和流程编排
- `src/` 下的模块负责各自独立的业务能力
- `tests/` 提供轻量测试，覆盖核心逻辑
- `samples/` 提供内置示例数据，用于演示和快速验证
- `docs/` 存放面向用户和面向评审的说明文档

整体上，它不是分布式架构，也不是严格意义上的前后端分离，而是一个围绕本地分析场景做的模块化单体应用。

### 2. 目录职责

- `app.py`：应用入口，负责 Streamlit 页面、侧边栏、按钮交互、分析触发和结果展示
- `src/loader.py`：负责文件识别与表格加载
- `src/validator.py`：负责单表校验
- `src/differ.py`：负责新旧表差异分析
- `src/llm_summary.py`：负责生成自然语言变更摘要
- `src/reporter.py`：负责把分析结果整理为可导出的报告
- `src/utils.py`：放置文件类型、扩展名、文件大小格式化等通用工具

### 3. 分层特点

这个项目虽然不大，但仍然刻意做了职责拆分，原因是为了避免把页面代码、数据逻辑和文本输出全部耦合在一起。这样做的好处有两个：

- 页面可以继续调整，而不影响核心分析逻辑
- 各模块可以单独测试，便于定位问题

这也是本项目“简单但不堆在一起”的实现原则。

## 四、核心流程

从当前代码看，主流程可以概括为“加载数据 -> 选择主键 -> 执行规则分析 -> 生成文字摘要 -> 导出报告”。

### 1. 数据进入系统

用户有两种方式进入分析流程：

- 上传自己的旧表和新表
- 在侧边栏切换内置示例，并自动加载

无论入口是哪一种，最终都会整理成统一的 `loaded_data` 结构，放入 `st.session_state`，后续分析流程共用同一套逻辑。

### 2. 文件加载

`app.py` 调用 `loader.load_table()` 读取文件内容，内部会先判断文件类型，再选择 CSV 或 Excel 的读取方式。

这一步不仅做“读取”，还做了第一层输入防御，例如：

- 文件是否为空
- 扩展名和内容是否匹配
- CSV 编码是否能识别
- Excel 是否损坏

如果这一层不通过，页面会直接给出友好的错误提示，而不是把异常暴露给用户。

### 3. 主键选择

加载完成后，系统会找出新旧两份表的共同列，并尝试推荐一个主键：

- 如果是内置示例，优先使用预设主键
- 如果是用户文件，则优先寻找在新旧表中都唯一的共同列

主键并不是一个可有可无的配置，它直接决定差异分析能否稳定进行。也因此，项目把“主键列选择”明确放在分析之前，而不是隐式处理。

### 4. 单表校验

点击“开始分析”后，程序先分别对旧表和新表调用 `validator.validate_table()`，生成两份结构一致的校验结果。

校验内容包括：

- 缺失值统计
- 重复主键检测
- 数值异常列识别
- 疑似类型不一致识别

这一步的输出不是给人直接阅读的长文本，而是结构化结果，为后面的差异分析、摘要生成和报告导出提供统一输入。

### 5. 差异分析

在单表校验完成后，程序调用 `differ.build_diff_summary()` 做新旧版本对比。

它会输出：

- 新增行
- 删除行
- 修改行
- 新增列
- 删除列
- 哪些字段变化最多
- 一共发生了多少单元格变化

这里最关键的设计点是：项目没有依赖模糊匹配，而是严格基于主键做行级对应。这样虽然牺牲了一些“自动猜测”的灵活性，但能换来更稳定、更可解释的结果。

### 6. 自然语言摘要生成

在完成规则分析后，用户可以按需点击“生成 AI 摘要”。程序会把结构化结果传给 `llm_summary.generate_change_summary()`。

这里有两种路径：

- 如果环境变量中配置了可用的模型接口，就调用外部 LLM
- 如果没有配置，或者调用失败，就回退到本地模板摘要

因此，这个功能不是“有模型才能工作”，而是“有模型时提升表达效果，没有模型时依然保证结果可交付”。

### 7. 报告导出

最后，`reporter` 模块会把分析结果和摘要拼装成报告文本。当前页面已提供 HTML 下载入口，代码层同时也支持 Markdown 报告生成与保存。

这保证了项目不仅能“分析出来”，还能“带着结果离开页面”。

## 五、核心模块职责说明

### 1. loader

`loader` 的职责不是简单地把文件读成 DataFrame，而是承担“输入规范化”的角色。

它负责：

- 识别文件类型
- 支持 CSV / XLS / XLSX 三类输入
- 处理 CSV 不同编码
- 拦截空文件、损坏文件、格式不匹配文件
- 输出统一的 DataFrame 结果

从设计上看，`loader` 是整个项目的第一道质量关口。只有把输入处理稳定，后续校验和对比逻辑才有意义。

### 2. validator

`validator` 负责单张表内部的质量检查，关注的是“这份表本身有没有问题”，而不是“新旧两份表有什么差异”。

当前实现覆盖了四类信号：

- 缺失值
- 重复主键
- 数值异常
- 疑似类型不一致

它的输出是结构化字典，其中同时包含明细和摘要。这使得同一份结果既能用于页面概览，也能用于详细展开。

### 3. differ

`differ` 负责新旧两版表格的差异比较，关注的是“版本变化”，而不是“数据质量”。

它的职责包括：

- 基于主键对齐新旧记录
- 判断新增行、删除行、修改行
- 判断新增列、删除列
- 统计字段变化次数
- 输出便于摘要和报告复用的差异结构

在整个系统中，`differ` 是连接“规则结果”和“业务说明”的核心模块，因为用户最关心的往往不是抽象统计，而是具体改了哪些内容。

### 4. llm_summary

`llm_summary` 负责把结构化分析结果转成自然语言说明。

它内部做了两件事：

- 组装提示词，把旧表校验、新表校验、Diff 结果统一交给模型
- 当模型不可用时，使用本地模板生成兜底摘要

这个模块的价值，不在于替代规则分析，而在于提升结果的可读性和交付感。它更像一个“结果表达层”，而不是“结果判断层”。

### 5. reporter

`reporter` 负责把前面各模块的输出整理成正式报告。

当前支持两种形式：

- Markdown
- HTML

它承担的职责包括：

- 整理文件信息
- 汇总关键指标
- 展示校验结果
- 展示 Diff 摘要
- 截取关键明细
- 插入 AI 或模板生成的变更说明

这一层让项目具备了更完整的交付形态，不只是一个页面工具，也是一份可沉淀的分析产物。

## 六、AI 在项目中的作用

本项目对 AI 的使用是克制而明确的。它的作用主要集中在“描述”和“总结”，而不是“判定”和“计算”。

### 1. AI 负责什么

AI 主要负责：

- 把结构化结果整理成自然语言摘要
- 帮用户快速形成一段适合沟通的变更说明
- 以更接近业务表达的方式呈现风险和建议

换句话说，AI 提升的是结果的表达效率，而不是分析的真实性。

### 2. 哪些判断仍由规则完成

以下关键判断都由规则逻辑完成，而不是交给模型：

- 文件类型识别
- 文件是否为空或损坏
- CSV/Excel 的加载逻辑
- 缺失值统计
- 重复主键判断
- 数值异常识别
- 疑似类型问题识别
- 新增行、删除行、修改行判断
- 新增列、删除列判断
- 变更字段统计

这些部分之所以坚持用规则完成，原因有三个：

1. 结果稳定
2. 逻辑可解释
3. 便于测试和回归验证

这也是本项目一个比较明确的设计原则：确定性的事情交给规则，不确定性的表达交给 AI。

### 3. 为什么这样设计

如果把“哪一行变了、有没有重复主键、某列是否异常”这类判断直接交给 AI，会带来两个问题：

- 结果不够稳定，难以复现
- 用户无法信任，也很难测试

因此，本项目把 AI 放在流程后段，只在规则结果已经明确后参与表达。这种边界划分既利用了 AI 的文本能力，也保留了工程实现的可靠性。

## 七、实现上的几个关键设计点

### 1. 以结构化结果为中心

无论是页面展示、AI 摘要还是报告导出，底层都尽量复用同一份结构化分析结果，而不是在多个地方重复计算。

这样做的好处是：

- 降低逻辑分叉
- 避免同一指标在不同地方口径不一致
- 方便测试和后续扩展

### 2. 页面流程与分析逻辑解耦

`app.py` 主要做流程组织，不直接写复杂分析逻辑。真正的文件加载、校验、Diff、摘要和报告都放在 `src/` 模块内。

这使项目即使以后更换前端壳，比如改成命令行或 FastAPI，也能复用现有核心逻辑。

### 3. 内置示例增强演示闭环

项目新增了内置示例场景，并支持侧边栏切换与自动加载。这一点对面试演示非常重要，因为它降低了“必须提前准备文件”的门槛，也让功能展示更稳定。

### 4. 有 AI，但不依赖 AI 才能跑通

项目的主流程不会因为没有模型配置而中断，这一点保证了工具在本地演示、离线评审和受限环境下依然可用。

从产品角度看，这种设计比“必须联网、必须有密钥”更稳妥。

## 八、测试与可维护性

虽然这是一个一天内完成的 MVP，但项目仍然补了轻量测试，覆盖的重点不是页面细节，而是最核心的业务逻辑：

- 文件读取
- 校验结果结构
- 差异比较结果
- 摘要生成回退逻辑
- 报告生成结果

这样的测试策略比较符合项目阶段：在有限时间内，优先守住最容易出错、最影响展示效果的部分。

同时，模块边界清楚也让后续扩展比较自然。例如后续如果要加入：

- 更多字段级规则
- 更丰富的风险分级
- 更强的 AI 提示词
- Markdown 下载按钮

都可以在现有结构上小步迭代，而不需要推翻重来。

## 九、项目总结

从实现角度看，这个项目并不追求“大而全”，而是围绕一个明确问题，完成了一条足够完整的产品链路：

- 能读文件
- 能发现问题
- 能比较版本
- 能生成说明
- 能导出结果

它的技术难点不在复杂算法，而在于如何在很短时间内把“规则校验、版本比对、AI 表达、页面交互、结果导出”组织成一个稳定、清晰、可演示的整体。

这也是本项目最想体现的工程价值：不是堆砌技术名词，而是在约束内做出一套边界清楚、体验完整、适合交付的解决方案。
